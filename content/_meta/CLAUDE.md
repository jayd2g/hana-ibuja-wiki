# _meta/CLAUDE.md — 아이부자 위키 운영 규칙

> **이 문서의 용도**: LLM(Claude) 및 팀원이 이 위키를 유지·확장할 때 따라야 할 구체적인 규칙.
> 개념적 가이드가 아니라 **실제 행동 지침**입니다.
>
> 기존 운영 규칙 원본: `content/CLAUDE.md` (유지됨)
> 이 파일은 원본의 확장판으로, 팀 협업 및 LLM 세대 콘텐츠 관리를 포함합니다.

---

## 1. 위키의 목적과 도메인 범위

**목적**: 하나은행 아이부자 UX 컨설팅 프로젝트의 리서치·분석·전략을 팀이 함께 축적하고 재활용할 수 있는 지식 베이스.

**도메인 범위 (이 위키에 속하는 내용)**:
- 청소년(알파~Z세대) 금융 행동, 발달심리, 금융사회화
- 한국 및 글로벌 청소년 금융 서비스 경쟁 구도 (토스유스, 카카오뱅크 미니, Greenlight, GoHenry 등)
- 아이부자 AS-IS 분석 및 To-Be 전략 설계
- 하나은행 아이부자팀 이해관계자 인터뷰 인사이트
- 부모-자녀 금융 상호작용, 발달단계별 UX 전략
- 아이부자 → 하나원큐 고객 생애주기 전략

**도메인 외 (이 위키에 넣지 않을 내용)**:
- 아이부자 개발/기술 구현 상세 (별도 기술 위키 필요)
- 하나은행 전체 사업 전략 (아이부자와 직접 관련이 없는 경우)
- 일반 UX 방법론 교과서적 설명 (이 프로젝트 맥락과 연결될 때만 포함)

## 2. 폴더 구조와 파일 배치 기준

```
content/
├── sc/              원본 자료 (PDF, docx, xlsx) — 절대 수정 금지
├── wiki/            정리된 지식 베이스 (Obsidian + Quartz로 퍼블리시)
│   ├── sources/     sc/ 자료 1건 = sources/ 1파일
│   ├── concepts/    개념·아이디어·프레임워크
│   ├── entities/    인물·조직·프로젝트
│   └── (root)       주제 요약 페이지 (시장/경쟁사/사용자/전략 요약)
├── Deskresearch/    분석 작업 문서 (카테고리별 심층 분석)
│   ├── 00_가설/
│   ├── 01_시장/
│   ├── 02_경쟁사/
│   ├── 03_사용자/
│   ├── 04_인사이트/
│   └── 05_Tobe전략/
├── output/          완성 산출물 (PPT, HTML, 차트, 보고서 등)
└── _meta/           위키 운영 메타 파일 (이 폴더)
    ├── CLAUDE.md    ← 이 파일
    ├── index.md     전체 문서 레지스트리
    └── taxonomy.md  태그 분류 체계
```

**파일 배치 판단 기준**:
- `sc/` → 원본 그대로 보관. 변환·요약 없음.
- `wiki/sources/` → sc/ 자료를 읽고 핵심 인사이트를 요약한 마크다운. 1 PDF = 1 sources 파일 (중복 금지).
- `wiki/concepts/` → 여러 소스에서 반복 등장하는 개념·아이디어. 단일 소스에서만 나오는 내용은 sources 파일 내 섹션으로 처리.
- `wiki/entities/` → 실명 인물(엄태성 상무 등), 특정 프로젝트, 조직.
- `Deskresearch/` → 분석·비교·종합 작업 문서. "인사이트를 도출하는 과정"이 담긴 파일.
- `output/` → 외부 발표용 PPT, 보고서 HTML, 차트 이미지 등 최종 산출물.

## 3. 파일명 규칙

### 한글 파일명 (wiki/root, 요약·허브 페이지)
- 공백 허용: `시장 분석 요약.md`, `용어 사전.md`
- 단, Obsidian 링크 시 `[[시장 분석 요약]]` 형태로 사용

### 영어+하이픈 파일명 (wiki/concepts, wiki/sources, wiki/entities)
- 소문자, 단어 구분은 하이픈: `금융적-성장-정의.md`, `F0401-의사결정권자-인터뷰.md`
- 날짜 포함 시: `아이부자_전략장표_내용_260414.md` (YYMMDD 형식)
- 공백 대신 하이픈, 언더스코어 모두 허용 (단, 파일 내에서 일관성 유지)

### Deskresearch/ 파일명
- 소문자+하이픈 또는 언더스코어: `부자가정_자녀금융교육_리서치.md`
- 날짜 포함 버전: `데이터보강_전략검증_260415.md`

## 4. Frontmatter 작성 규칙

모든 `.md` 파일은 YAML frontmatter로 시작해야 합니다.

### 전체 스키마
```yaml
---
title: "파일 제목 (한국어 권장)"
type: source | concept | entity | topic | analysis | navigation | log | glossary | reference
tags: [태그1, 태그2]
date_updated: YYYY-MM-DD
author_type: human | llm | mixed
provenance: extracted | inferred | ambiguous
---
```

### 각 필드 작성 기준

**`type`**
| 값 | 위치 | 설명 |
|---|---|---|
| `source` | `wiki/sources/` | 원본 자료(PDF 등) 요약 |
| `concept` | `wiki/concepts/` | 개념·아이디어·프레임워크 |
| `entity` | `wiki/entities/` | 인물·프로젝트·조직 |
| `topic` | `wiki/` 루트 | 주제 요약·개요 페이지 |
| `analysis` | `Deskresearch/` | 분석·종합 작업 문서 |
| `navigation` | HOME.md, index.md | 탐색 허브 |
| `log` | log.md | 활동 로그 |
| `glossary` | 용어 사전.md | 용어 정의집 |
| `reference` | 참고문헌 마스터.md | 문헌 목록 |

**`author_type`**
- `llm`: Claude 등 AI가 작성 (대부분의 wiki 파일)
- `human`: 팀원이 직접 작성
- `mixed`: 일부는 LLM, 일부는 사람이 작성·수정

**`provenance`**
- `extracted`: 원본 자료에서 직접 추출 (sources/, entities/ 인터뷰 요약)
- `inferred`: 여러 소스를 종합해 LLM이 추론·합성 (concepts/, Deskresearch/)
- `ambiguous`: 출처 불명확, 추론과 인용의 경계가 모호한 경우

**`date_updated`**: 마지막으로 내용이 의미있게 변경된 날짜 (YYYY-MM-DD). 파일 저장 날짜가 아닌 내용 갱신 날짜.

## 5. [[Wikilink]] 연결 기준

### 링크를 걸어야 하는 경우
- 같은 위키 내 다른 페이지에 있는 개념이 본문에 처음 등장할 때
- 핵심 판단의 근거가 되는 sources 페이지 참조
- 인물·조직이 본문에서 행위의 주체로 등장할 때

### 링크를 걸지 않는 경우
- 같은 파일 내에서 반복 등장하는 단어 (첫 등장에만 걸 것)
- 위키에 페이지가 없는 개념 (없다면 페이지를 먼저 만들거나 그냥 plain text로)
- 목차·제목 줄
- "아이부자", "하나은행", "청소년" 같은 프로젝트 전반의 기본 용어 (너무 자주 등장)

### 링크 형식
```
[[concepts/금융적-성장-정의]]          # 폴더 포함 풀 경로 (권장)
[[금융적-성장-정의]]                   # 파일명만 (Obsidian에서 자동 해석)
[[concepts/금융적-성장-정의|성장 기준]]  # 표시 텍스트 변경
```

## 7. 새 페이지 생성 vs 기존 페이지 업데이트 기준

### 새 페이지를 만들어야 할 때
- 위키에 아직 없는 개념이고, 최소 2개 이상의 소스에서 독립적으로 등장할 때
- 새 원본 자료(PDF, 인터뷰 등)를 처음 처리할 때 (`wiki/sources/` 신규)
- 새 인물이나 조직이 프로젝트에서 중요한 역할을 할 때 (`wiki/entities/` 신규)

### 기존 페이지를 업데이트해야 할 때
- 이미 있는 개념에 새 데이터·출처·반례가 추가될 때
- 기존 sources 페이지의 해석이 바뀌거나 오류가 발견됐을 때
- `source_count` 값이 늘어날 때 (기존 concepts 페이지 업데이트 + log에 기록)

### 판단이 애매한 경우
- 유사 개념이 이미 있다면 → 기존 페이지에 섹션 추가 후 판단
- 확실히 독립적 개념이라 생각되면 → 새 페이지 만들고 기존 페이지에서 링크

## 8. LLM 생성 vs 사람 작성 콘텐츠 구분

### frontmatter로 구분
```yaml
author_type: llm     # Claude가 생성
author_type: human   # 팀원이 직접 작성
author_type: mixed   # 혼합
```

### 본문 내 구분 (선택적)
LLM이 추론한 내용 중 불확실한 부분은 callout으로 표시:
```markdown
> [!warning] LLM 추론
> 이 해석은 데이터에서 직접 확인된 것이 아니라 소스들을 종합해 추론된 것입니다.
> 검증 전까지 `provenance: ambiguous`로 처리합니다.
```

### 인용 표기
원본 자료의 특정 수치나 문장은 출처를 명시:
```markdown
부모 영향이 금융사회화에서 가장 강한 직접효과를 보임 (B=0.554, 조선영·최현자, 2023)
```

## 9. 팀원 간 충돌 처리 (모순 발생 시)

### 데이터 충돌 (수치·사실이 다를 때)
1. 두 출처를 모두 기록하고 날짜·방법론 차이를 명시
2. 더 최신·더 대규모 조사를 우선
3. 판단 불가 시 `provenance: ambiguous` 표시 + `_meta/index.md`의 "검토 필요" 목록에 추가

### 해석 충돌 (LLM 추론 vs 팀원 판단이 다를 때)
1. 팀원 판단이 우선 → `author_type: mixed`로 변경, 변경 내용 log.md에 기록
2. 원래 LLM 추론 내용은 삭제하지 않고 인용 블록으로 보존:
```markdown
> [!note] 기존 LLM 추론 (2026-04-10, 팀 검토 후 수정됨)
> [기존 내용]
```

### 전략 방향 충돌 (이전 전략안 vs 새 전략안)
- 이전 전략안은 삭제하지 않고 `Deskresearch/05_Tobe전략/` 내에 날짜 버전으로 보존
- `wiki/` 내 요약 페이지는 최신 전략만 반영하되, 이전 버전 링크 유지

## 10. 금지 사항

| 금지 행위 | 이유 |
|---|---|
| `sc/` 내 파일 수정·삭제 | 원본 보존 원칙. 원본이 바뀌면 모든 sources 파일 재작성 필요 |
| `log.md` 기존 항목 수정 | append-only. 오류 수정은 새 항목으로 추가 (`## [날짜] correction | ...`) |
| 동일 PDF에 sources 파일 중복 생성 | ingest 전 반드시 `wiki/index.md` 확인 |
| `_meta/` 파일을 운영 목적 외 내용으로 채우기 | 운영 메타 폴더이므로 지식 콘텐츠는 `wiki/`로 |
| `output/` 파일을 wiki 페이지에서 직접 링크 | 배포 시 경로가 달라질 수 있음; output/ 내 파일은 별도 경로로 접근 |
| Harvard ALI "60~70% 재투자" 수치 인용 | 출처 불명확으로 판명됨 (2026-04-15 log 참조). 대체: SEED OK RCT 사용 |

## 11. 워크플로우 요약

### Claude Code / Cowork 에서 새 자료 처리 (Ingest)
```
1. sc/ 에 파일 존재 확인
2. wiki/index.md 에서 동일 자료 기처리 여부 확인
3. 자료 읽기 → 핵심 인사이트 정리
4. wiki/sources/[파일명].md 작성 (frontmatter 포함)
5. 관련 wiki/concepts/ 업데이트 또는 신규 생성
6. wiki/index.md 업데이트
7. wiki/log.md 에 항목 추가
8. _meta/index.md 의 "문서 레지스트리" 업데이트
```

### Obsidian 에서 팀원 직접 편집 시
```
1. author_type: human 으로 변경 (또는 mixed)
2. date_updated: 오늘 날짜 수정
3. 변경 내용 간략히 log.md 에 추가
4. git commit (obsidian-git 플러그인 자동 또는 수동)
```

### GitHub에서 PR로 협업 시
```
1. feature 브랜치 생성: wiki/[주제]-[날짜]
2. 변경 내용 작성 후 PR
3. 리뷰어: 내용 충돌 여부, frontmatter 완성도 확인
4. merge 후 Quartz 자동 빌드 (GitHub Actions)
```
